# Secure Multi-Tenant Payroll Infrastructure (Terraform)

## What this project does

This repository provisions an AWS-based, production-oriented infrastructure for a multi-tenant payroll platform using Terraform. The implementation in this codebase provides:

- A VPC with public and private subnets across two Availability Zones.
- Tenant-specific compute (EC2 instances) created per tenant name provided in `tenant_names`.
- A private, encrypted PostgreSQL database (RDS) and a Secrets Manager secret that stores DB credentials.
- A single S3 bucket for tenant document storage with prefix-based isolation, versioning, and SSE-KMS encryption.
- Tenant-specific IAM roles and instance profiles with least-privilege policies.
- Network controls (security groups and VPC endpoints) and infrastructure outputs to integrate services.

This README documents exactly what the Terraform code implements and how the pieces connect.

## Repository modules and responsibilities

- `modules/vpc` — creates the VPC, public and private subnets, NAT gateway(s), route tables, NACL placeholders and VPC endpoints (including an S3 Gateway endpoint). Exposes `vpc_id`, `public_subnet_ids`, and `private_subnet_ids`.
- `modules/tenants` — creates one security group per tenant and EC2 instances per tenant using `tenant_names`. Instances use instance profiles supplied by the IAM module and are placed in the private subnets. Instances are created without public IPs.
- `modules/iam` — provisions tenant IAM roles, tenant IAM policies scoped to S3 prefixes and Secrets Manager, and instance profiles. It exposes `role_arns`, `role_names`, and `instance_profile_names`.
- `modules/storage` — provisions a single S3 bucket (name includes a random suffix), enables versioning and server-side encryption (SSE-KMS), blocks public access, and attaches an explicit bucket policy that grants each tenant access only to their prefix. Inputs include `bucket_name`, `kms_key_id`, `tenants`, `tenant_role_arns`, and `tags`.
- `modules/database` — creates a DB subnet group, a DB security group, a generated password, a Secrets Manager secret containing JSON `{username,password}`, and an RDS instance that reads the password from the created secret. Outputs include `db_secret_arn` and `db_instance_id`.

## How components are connected (implementation detail)

- The root module wires resources together by passing `module.vpc.private_subnet_ids` into `modules/tenants` and `modules/database`.
- `modules/iam` creates tenant roles and instance profiles; `modules/tenants` consumes `module.iam.instance_profile_names` to attach profiles to EC2 instances.
- `modules/storage` receives `module.iam.role_arns` from the IAM module and `var.tenant_names` from the root to construct a bucket policy granting each tenant access to `arn:aws:s3:::<bucket>/<tenant>/*` and conditional `ListBucket` restricted to the tenant prefix.
- The database module generates a password with `random_password`, stores credentials in `aws_secretsmanager_secret` + `aws_secretsmanager_secret_version`, and sets the RDS `password` by decoding the secret version (`jsondecode(...)["password"]`). The secret ARN is exported as `db_secret_arn` (the secret value is not exported).

## Multi-tenancy design (concrete)

- Tenants are defined via the root variable `tenant_names` (e.g., `["company","bureau","employee"]`). The same list is passed to `modules/tenants` and `modules/storage`.
- Compute isolation: `modules/tenants` creates a dedicated security group per tenant and attaches EC2 instances to that SG. EC2 instances are placed in private subnets and use per-tenant instance profiles.
- Storage isolation: `modules/storage` enforces prefix-based isolation using a bucket policy that only allows each tenant role ARN (from `module.iam.role_arns`) to act on their prefix. Actions allowed include `s3:GetObject`, `s3:PutObject`, `s3:DeleteObject`, and `s3:ListBucket` (conditioned by prefix).
- IAM isolation: `modules/iam` provides roles that grant only the permissions required by tenants — S3 prefix-level actions, Secrets Manager read for the specific DB secret ARN, and `ec2:DescribeInstances` as implemented.

## Security implementation (exact)

- Secrets management
  - A Secrets Manager secret is created by `modules/database` and contains JSON `{ "username": "...", "password": "..." }` where the password is generated by `random_password` in the same module.
  - The DB RDS password is read from the Secrets Manager secret version; the secret value is not written as a root-level Terraform output.

- Encryption
  - S3: bucket uses SSE-KMS with the configured KMS key id (the project uses `data.aws_kms_alias.s3` or module-provided KMS depending on root wiring). Versioning is enabled.
  - RDS: storage is encrypted and the database references the `kms_key_id` variable when provided.

- Network and access
  - EC2 instances are launched without public IPs and rely on SSM for access via instance profile attachments.
  - RDS is `publicly_accessible = false`, placed in private subnets, and a DB security group allows ingress only from tenant security groups.
  - VPC includes an S3 Gateway endpoint so S3 traffic can remain on the AWS network.

- IAM least privilege
  - Tenant IAM policies contain explicit resource ARNs (no wildcard `*` for tenant-scoped resources) for S3 and restrict Secrets Manager access to the `db_secret_arn`.
  - The `AmazonSSMManagedInstanceCore` managed policy is attached via instance profiles to support SSM session manager and secure instance management.

## Real data flow (based on code)

1. Application (running on an EC2 tenant instance) requests DB credentials from Secrets Manager using its tenant role (allowed by a least-privilege IAM statement that names the secret ARN).
2. The application uses the credential to connect to RDS over the private subnet and defined DB security group.
3. The application reads/writes tenant documents to S3 under the tenant-specific prefix (e.g., `s3://<bucket>/company/...`). The bucket policy enforces this prefix-level access.

## Deployment steps (practical and exact)

1. Configure backend: edit `aws_multi_tenant/backend.tf` with your S3 backend and DynamoDB table for locks (secure these values; do not commit secrets).
2. Supply environment variables or `terraform.tfvars` in `environments/dev` or `environments/prod` for values such as `project_name`, `tenant_names`, `bucket_name`, and `db_secret_arn` where required.
3. Initialize and validate:
```bash
cd aws_multi_tenant
terraform init
terraform validate
```
4. Plan and apply (example dev):
```bash
terraform plan -var-file="../environments/dev/terraform.tfvars"
terraform apply -var-file="../environments/dev/terraform.tfvars"
```

Notes on secrets and DB creation:
- The current database module creates a Secrets Manager secret and RDS instance using the generated password. If you prefer not to create credentials via Terraform, create an external secret and provide its ARN as `db_secret_arn` to root (the code supports reading an existing secret in some configurations). Protect state and restrict backend access.

## Outputs (what the code exposes)

- `module.vpc.vpc_id` — VPC identifier used by other modules.
- `module.vpc.private_subnet_ids` — Private subnet IDs for placing EC2 and RDS.
- `module.storage.bucket_arn` / `module.storage.bucket_name` — S3 bucket identifiers for application configuration.
- `module.iam.role_arns` — Map of tenant → IAM role ARN; used to generate S3 bucket policy principals.
- `module.iam.instance_profile_names` — Instance profile names to attach to EC2 instances.
- `module.database.db_secret_arn` — ARN of the Secrets Manager secret holding DB credentials (the secret value is not exported).
- `module.database.db_instance_id` — RDS instance id if the DB was created by the module.

## What is implemented vs. intentionally omitted

- Implemented: Tenant-specific IAM roles, S3 prefix isolation, Secrets Manager integration, private EC2/RDS placement, S3 SSE-KMS, and VPC S3 endpoint.
- Omitted or left for operator decision: centralized logging destinations beyond basic CloudWatch resources, full CI/CD secret provisioning (a GitHub Actions workflow template is present but requires organization-specific secrets), and automated secret rotation for DB credentials.

## Interview talking points (how to present this project)

- Explain the multi-tenant model: a single account with logical isolation via IAM + SG + S3 prefix policies; mention trade-offs (operational simplicity vs. absolute isolation).
- Describe secrets handling: DB credentials are stored in Secrets Manager and used by RDS; Terraform avoids printing secret values in outputs.
- Describe network security: private subnets for compute and DB, no public IPs on instances, VPC endpoint for S3 to keep traffic internal.
- Explain IAM least-privilege: tenant roles only get the ARNs that match their prefixes and the specific Secret ARN.
- Mention operations: use SSM Session Manager to access instances; use remote backend with encryption and locking for team safety.

## Next steps & improvements (practical)

- Add CloudWatch alarms and SNS integration for alerting (some alarm resources are present; customize thresholds and notification targets).
- Integrate CI for provisioning and secret injection using an out-of-band bootstrap if production rules forbid Terraform-managed secrets.
- Harden KMS key policies to explicitly list principals and services that can use the key.
- Consider database row-level security or separate databases per tenant if stronger data isolation is required.
